# Fixture: py-inventory-service (Python)

[panopticon-test-child-a](https://github.com/industrial-curiosity/panopticon-test-child-a)

Test fixture repo representing a Python inventory and fulfillment service. Covers: internal-only
interfaces, externally owned interfaces (external SaaS and managed infrastructure), and an
in-repo interface-naming collision (`orders-api` self-claimed alongside an existing consumed
declaration of the same name).

## Purpose

Exercises Panopticon against a Python repo where every interface is extracted via LLM fallback
directly from `.py` source files — there is no `infra/` directory and no deterministic
REST/OpenAPI or Kafka parser input in this repo:

- All interfaces (including the one backed by an `openapi.yaml` contract) are recorded with
  `extracted_by: "llm"` in `panopticon/index.json` — none are picked up by a deterministic parser.
- A `panopticon-interface <name>` hint comment pins the canonical name only where one is present
  in source (`inventory/api/orders_routes.py`); interfaces without a hint are named by LLM
  judgment instead.
- Several interfaces have `null` owner (external ERP, managed database, Kafka topic, and an
  order-processing status endpoint — none owned by any org repo, per this repo's own index).
- `inventory/api/orders_routes.py` declares an inventory-owned `orders-api`, the same canonical
  name and type already declared as consumed (not owned) via `inventory/clients/orders.py` —
  an unreconciled ownership dispute visible entirely within this repo's own index (both a
  producer and a consumer entry for the same name/type, locally).

## Interfaces

### inventory-api — owned, no cross-repo consumer visible locally

- **Type:** `rest`
- **Owner:** `panopticon-test-child-a / api`
- **Producer sources:** `inventory/api/openapi.yaml`, `inventory/api/routes.py`
- **Consumers (other repos):** none declared in this repo's own index
- **Provenance:** LLM fallback (no `panopticon-interface` hint present in either file)
- **Category:** owned by this repo; whether a sibling repo consumes it is not visible from here

### orders-api — owned here AND consumed here (naming collision)

- **Type:** `rest`
- **Owner:** `panopticon-test-child-a / api`
- **Producer source:** `inventory/api/orders_routes.py` (pinned by a
  `# panopticon-interface orders-api` hint comment)
- **Consumer source:** `inventory/clients/orders.py` (same repo)
- **Provenance:** LLM fallback
- **Category:** unreconciled — this repo's own index both produces and consumes the identical
  canonical name and type; `orders_routes.py` is an in-progress, inventory-owned replacement for
  the API `clients/orders.py` still calls externally. This is a real local collision, not a
  cross-repo one — everything needed to see it is in this repo's own index.

### inventory-snapshots — internal only

- **Type:** `s3`
- **Owner:** `panopticon-test-child-a / storage`
- **Source:** `inventory/storage/snapshots.py`
- **Consumers (other repos):** none
- **Provenance:** LLM fallback
- **Category:** internal; daily snapshot bucket for audit/recovery, never shared across repos

### fulfillment-queue — internal only

- **Type:** `sqs`
- **Owner:** `panopticon-test-child-a / queue`
- **Source:** `inventory/queue/fulfillment_queue.py`
- **Consumers (other repos):** none
- **Provenance:** LLM fallback
- **Category:** internal; async fulfillment task queue, never crossed to another repo

### warehouse-erp — external service, manually managed

- **Type:** `rest`
- **Owner:** `null` (third-party on-premise ERP; no org repo owns or can modify it)
- **Source:** `inventory/clients/erp.py`
- **Category:** external; contract managed by ops team outside of version control

### order-processing-queue — external, owner unresolved

- **Type:** `rest`
- **Owner:** `null` in this repo's index
- **Source:** `inventory/clients/order_processing.py`
- **Category:** external; this repo has no visibility into whether another repo declares this
  name under a different type — that can only be seen from the compiled instance index, not from
  this shard

### order-events — external to this repo

- **Type:** `kafka`
- **Owner:** `null` in this repo's index
- **Source:** `inventory/events/kafka_consumer.py`
- **Category:** external; truth about ownership, if any, lives outside this repo

### product-catalog-db — managed infrastructure, no code owner

- **Type:** `database`
- **Owner:** `null` (managed database instance; no code in this repo provisions it)
- **Source:** `inventory/db/catalog.py`
- **Category:** external/infrastructure; connection details come from an environment variable
  (`CATALOG_DB_DSN`), not from any org-owned service

## Panopticon test coverage

| Scenario | Interface | Detail |
|---|---|---|
| LLM fallback, no hint | `inventory-api` | No parser input recorded (`openapi.yaml` present but `extracted_by: "llm"` regardless); named by LLM judgment over both files |
| LLM fallback + hint | `orders-api` | `# panopticon-interface orders-api` in `orders_routes.py` pins the name |
| LLM fallback, no hint | `inventory-snapshots`, `fulfillment-queue`, `warehouse-erp`, `order-processing-queue`, `order-events`, `product-catalog-db` | Named by LLM judgment; no hint comment present |
| Internal-only | `inventory-snapshots`, `fulfillment-queue` | Owner = this repo; no other repo's index references them |
| External / null owner | `warehouse-erp`, `order-processing-queue`, `order-events`, `product-catalog-db` | No org repo owns any of these in this repo's index |
| In-repo naming collision | `orders-api` | This repo's index both produces (`orders_routes.py`) and consumes (`clients/orders.py`) the identical canonical name and type — an unreconciled ownership dispute visible entirely from this one shard, not a cross-repo conflict |
